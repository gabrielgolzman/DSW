# Introducción al Código Servidor

## ¿Qué es el Backend?

Una aplicación web moderna se divide en tres capas que trabajan en conjunto:

![3 layers web dev](im%C3%A1genes/3_layers_web_dev.png)

- **Frontend (cliente):** Lo que el usuario ve y con lo que interactúa en el navegador. Está construido con HTML, CSS y JavaScript, y corre en el navegador del usuario.
- **Backend (servidor):** La lógica de negocio de la aplicación. Recibe pedidos del frontend, los procesa, y devuelve una respuesta. El usuario nunca lo ve directamente.
- **Base de datos:** Persiste la información. El backend es el único que se comunica con ella directamente.

La comunicación entre capas sigue este flujo:

```
[Navegador / Frontend]
        |
        |  HTTP Request (ej: "dame todos los libros")
        ↓
[Servidor / Backend]
        |
        |  Consulta SQL / NoSQL
        ↓
[Base de datos]
        |
        ↑  Datos
        |
[Servidor / Backend]
        |
        |  HTTP Response (JSON con los libros)
        ↑
[Navegador / Frontend]
```

El frontend y el backend se comunican mediante **HTTP**: el frontend envía un *request* con un verbo y una ruta, y el backend responde con un *status code* y, generalmente, datos en formato **JSON**.

## Configurando Express

### Inicializar el proyecto

Todo proyecto de Node.js comienza con:

```bash
npm init -y
```


`npm init -y` genera el archivo **`package.json`**, que es el manifiesto del proyecto: describe sus metadatos y lista sus dependencias.

```json
{
  "name": "book-champions-api",
  "version": "1.0.0",
  "description": "",
  "license": "ISC",
  "author": "",
  "type": "module",
  "main": "app.ts",
  "scripts": {
    "dev": "tsc-watch --noClear -p ./tsconfig.json --onSuccess \"node ./dist/app.js\""
  },
  "dependencies": {
    "express": "^5.2.1"
  },
  "devDependencies": {
    "@types/express": "^5.0.6",
    "@types/node": "^25.9.3",
    "tsc-watch": "^7.2.0",
    "typescript": "^6.0.3"
  }
}
```

- **`"type": "module"`** — indica que el proyecto usa ES Modules (`import`/`export`) en lugar de CommonJS (`require`).
- **`dependencies`** — paquetes necesarios en producción (ej: `express`).
- **`devDependencies`** — paquetes solo necesarios durante el desarrollo (ej: TypeScript y sus tipos).

### Instalar dependencias

```bash
npm install express
npm install -D typescript tsc-watch @types/express @types/node
```

Esto descarga los paquetes en la carpeta `node_modules/` y genera el **`package-lock.json`**.

El **`package-lock.json`** congela las versiones exactas de cada paquete instalado (incluyendo dependencias de dependencias). Sirve para garantizar que todos los integrantes del equipo instalen exactamente las mismas versiones al hacer `npm install`.

> **Importante:** `node_modules/` nunca se commitea al repositorio. Se agrega al `.gitignore`. Con el `package.json` y el `package-lock.json` alcanza para que cualquiera pueda recrear el proyecto con `npm install`.

### Scripts

En `package.json` definimos comandos reutilizables bajo `"scripts"`:

```json
"scripts": {
  "dev": "tsc-watch --noClear -p ./tsconfig.json --onSuccess \"node ./dist/app.js\""
}
```

Se ejecutan con `npm run <nombre>`. En este caso: `npm run dev`.

**¿Qué hace este script?**

- **`tsc-watch`** — vigila los archivos `.ts` en busca de cambios y recompila automáticamente cuando detecta uno.
- **`--noClear`** — evita limpiar la consola en cada recompilación.
- **`-p ./tsconfig.json`** — le dice al compilador qué configuración de TypeScript usar.
- **`--onSuccess "node ./dist/app.js"`** — una vez que la compilación termina con éxito, levanta el servidor ejecutando el archivo JavaScript compilado.

El resultado: cada vez que modificamos un `.ts` y guardamos, el servidor se recompila y reinicia automáticamente.

### tsconfig.json

El archivo **`tsconfig.json`** configura el compilador de TypeScript:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "rootDir": "./src",
    "outDir": "./dist",
    "sourceMap": true,
    "strict": true,
    "esModuleInterop": true,
    "forceConsistentCasingInFileNames": true,
    "skipLibCheck": true,
    "incremental": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

| Opción | Qué hace |
|---|---|
| `target` | Define la versión de JavaScript a la que se compila el código TypeScript (`ES2022` es compatible con Node.js moderno). |
| `module` | Especifica el sistema de módulos del código compilado (`NodeNext` usa ES Modules nativos de Node). |
| `moduleResolution` | Define cómo el compilador busca y resuelve los módulos importados. |
| `rootDir` | Carpeta raíz donde viven los archivos fuente `.ts`. |
| `outDir` | Carpeta donde el compilador deposita los archivos `.js` generados. |
| `sourceMap` | Genera archivos `.map` que vinculan el `.js` compilado con el `.ts` original, facilitando el debugging. |
| `strict` | Activa todas las verificaciones de tipo estrictas de TypeScript para detectar errores en tiempo de compilación. |
| `esModuleInterop` | Permite importar módulos CommonJS (como `express`) con la sintaxis de ES Modules (`import express from 'express'`). |
| `forceConsistentCasingInFileNames` | Obliga a que las rutas de importación respeten el mismo uso de mayúsculas/minúsculas que el nombre real del archivo. |
| `skipLibCheck` | Omite la verificación de tipos en los archivos `.d.ts` de `node_modules` para acelerar la compilación. |
| `incremental` | Guarda un caché de la compilación anterior para recompilar solo lo que cambió. |

### Nuestro primer endpoint: `app.ts`

```typescript
import express from "express";

const app = express();

const PORT = 3000;

app.use(express.json());

app.get("/", (req, res) => {
    res.send("You reached the App!");
});

app.listen(PORT, () => {
    console.log(`Server listening in ${PORT}`);
});
```

**¿Qué hace cada parte?**

- **`import express from "express"`** — importa el framework Express.
- **`const app = express()`** — crea la aplicación, el objeto central que recibirá los requests.
- **`app.use(express.json())`** — registra un *middleware*. Un **middleware** es una función que se ejecuta en cada request antes de llegar al endpoint, y puede transformar o inspeccionar el request/response. En este caso, parsea automáticamente el body en formato JSON.
- **`app.get("/", (req, res) => {...})`** — define un endpoint. El callback recibe dos objetos clave:
  - **`req`** (Request): contiene todo sobre el pedido entrante — headers, body, parámetros de ruta, query params.
  - **`res`** (Response): el objeto para construir y enviar la respuesta al cliente.
- **`app.listen(PORT, ...)`** — levanta el servidor y lo pone a escuchar en el puerto 3000.

Al ejecutar `npm run dev`, deberías ver en la terminal:

```
Server listening in 3000
```

Eso confirma que el servidor está corriendo. Podés probarlo abriendo `http://localhost:3000` en el navegador o con una herramienta como [Postman](https://www.postman.com/), [Thunder Client](https://www.thunderclient.com/) o [Bruno](https://www.usebruno.com/).

### ¿Qué es una REST API?

Una **REST API** (Representational State Transfer) es una convención para diseñar endpoints HTTP. En lugar de inventar nombres de acciones (`/getBooks`, `/deleteBook`, `/createBook`), REST aprovecha los **verbos HTTP** y las **rutas** para expresar la intención:

> Un endpoint REST = **verbo HTTP** + **ruta**

Por ejemplo:
- `GET /books` → traer todos los libros
- `POST /books` → crear un nuevo libro
- `GET /books/42` → traer el libro con id 42
- `PUT /books/42` → reemplazar el libro con id 42
- `DELETE /books/42` → eliminar el libro con id 42

La ruta identifica el **recurso** (libro, usuario, etc.) y el verbo indica la **acción** sobre ese recurso.

---

### Verbos HTTP y sus usos

| Verbo | Uso | Tiene body? |
|---|---|---|
| **GET** | Obtener uno o más recursos. No modifica datos. | No |
| **POST** | Crear un nuevo recurso. Los datos van en el body del request. | Sí |
| **PUT** | Reemplazar un recurso completo. Se envía el recurso entero actualizado. | Sí |
| **PATCH** | Modificar parcialmente un recurso. Solo se envían los campos que cambian. | Sí |
| **DELETE** | Eliminar un recurso. | No |

En Express, cada verbo tiene su método correspondiente:

```typescript
app.get("/books", (req, res) => { /* traer libros */ });
app.post("/books", (req, res) => { /* crear libro */ });
app.put("/books/:id", (req, res) => { /* reemplazar libro */ });
app.patch("/books/:id", (req, res) => { /* modificar libro */ });
app.delete("/books/:id", (req, res) => { /* eliminar libro */ });
```

El parámetro `:id` en la ruta es un **parámetro de ruta** (*route param*) que Express captura y pone disponible en `req.params.id`.

## Implementando los endpoints: Book Champions API

Veamos cómo se implementa cada endpoint en un proyecto real. La aplicación maneja una colección de libros guardada en memoria (un array `BOOKS_MOCK`).

### Modelo de datos

Antes de ver los endpoints, el objeto central es la clase `Book`:

```typescript
import crypto from "node:crypto"

export class Book {
    constructor(
        public title: string,
        public authors: string[],
        public publisher: string,
        public pageCount: number,
        public rating: number,
        public cover: string,
        public isAvailable: boolean,
        public id = crypto.randomUUID()
    ) { }
}
```

`crypto.randomUUID()` genera un identificador único universal (UUID) para cada libro. Es parte del módulo nativo `node:crypto`, por lo que no requiere instalar ningún paquete externo. El prefijo `node:` es la forma moderna de importar módulos nativos de Node.js y evita ambigüedades con paquetes de npm que tengan el mismo nombre.

---

### `GET /api/books` — Obtener todos los libros

```typescript
app.get("/api/books", (req, res) => {
    res.json(BOOKS_MOCK);
});
```

El endpoint más simple: devuelve todos los libros del array. `res.json()` serializa el array a JSON y automáticamente setea el header `Content-Type: application/json` en la respuesta.

---

### `GET /api/books/:id` — Obtener un libro por ID

```typescript
app.get("/api/books/:id", (req, res) => {
    const bookIndex = BOOKS_MOCK.findIndex((book) =>
        book.id === req.params?.id);

    if (bookIndex < 0)
        return res.status(404).send({ message: "Book not found" });

    return res.json(BOOKS_MOCK[bookIndex]);
});
```

Este endpoint introduce dos conceptos nuevos:

- **`req.params.id`** — captura el valor que viene en la URL. Si el cliente hace `GET /api/books/pepito`, entonces `req.params.id` vale `"pepito"`.
- **`findIndex()`** — busca en el array y devuelve el índice del primer elemento que cumple la condición, o `-1` si no lo encuentra.

Si el libro no existe, respondemos con `status(404)` (*Not Found*). Es importante **retornar** en ese punto para que la función no siga ejecutándose y no intente enviar una segunda respuesta.

---

### `POST /api/books` — Crear un libro

```typescript
app.post("/api/books", sanitizeBookInput, (req, res) => {
    const input = req.body.sanitizedBookInput;

    const book = new Book(
        input.title,
        input.authors,
        input.publisher,
        input.pageCount,
        input.rating,
        input.cover,
        input.isAvailable,
    );

    BOOKS_MOCK.push(book);

    return res.json(book);
});
```

Acá aparece `sanitizeBookInput` entre la ruta y el handler: es un **middleware** que se ejecuta primero y limpia los datos del request antes de que lleguen al handler principal. Lo explicaremos mejor en la próxima sección.

El handler toma los datos ya limpios de `req.body.sanitizedBookInput`, construye una instancia de `Book` (que genera su propio `id` con `randomUUID()`), la agrega al array y la devuelve al cliente.

---

### `PUT /api/books/:id` — Reemplazar un libro completo

```typescript
app.put("/api/books/:id", sanitizeBookInput, (req, res) => {
    const bookIndex = BOOKS_MOCK.findIndex((book) => book.id === req.params.id);

    if (bookIndex < 0)
        return res.status(404).send({ message: "Book not found" });

    BOOKS_MOCK[bookIndex] = {
        ...BOOKS_MOCK[bookIndex],
        ...req.body.sanitizedBookInput
    };

    res.json(BOOKS_MOCK[bookIndex]);
});
```

`PUT` reemplaza un recurso completo. La estrategia es usar el **spread operator** (`...`): primero se copia el libro existente y luego se sobreescriben sus propiedades con los datos de la request. Esto preserva el `id` original (que no viene en el body) mientras actualiza todos los demás campos.

---

### `PATCH /api/books/:id` — Modificar parcialmente un libro

```typescript
app.patch('/api/books/:id', sanitizeBookInput, (req, res) => {
    const bookIdx = BOOKS_MOCK.findIndex((book) => book.id === req.params.id);

    if (bookIdx < 0) {
        return res.status(404).send({ message: 'Book not found' });
    }

    Object.assign(BOOKS_MOCK[bookIdx], req.body.sanitizedBookInput);

    return res.status(200).send({ message: 'Book updated successfully', data: BOOKS_MOCK[bookIdx] });
});
```

`PATCH` solo actualiza los campos que el cliente envía. La diferencia clave con `PUT`:

- **`PUT`** reemplaza el objeto completo (el cliente envía todos los campos).
- **`PATCH`** fusiona solo los campos recibidos sobre el objeto existente.

`Object.assign(target, source)` copia las propiedades de `source` sobre `target`, modificando el objeto original en el array directamente. Gracias a `sanitizeBookInput`, los campos que no vinieron en el request ya fueron eliminados del body, así que `Object.assign` no sobreescribe nada que no debería.

---

### `DELETE /api/books/:id` — Eliminar un libro

```typescript
app.delete("/api/books/:id", (req, res) => {
    const bookIndex = BOOKS_MOCK.findIndex((book) => book.id === req.params.id);

    if (bookIndex < 0)
        return res.status(404).send({ message: "Book not found" });

    BOOKS_MOCK.splice(bookIndex, 1);

    return res.json({ message: `Book with id: ${req.params.id} successfully deleted` });
});
```

`Array.splice(index, 1)` elimina un elemento en una posición dada y muta el array original. Como el libro ya no existe, la respuesta devuelve un mensaje de confirmación en lugar del recurso eliminado.

## Sanitización de entrada

### ¿Por qué sanitizar?

Cuando un cliente envía datos en el `body` de un request, el servidor no puede asumir que esos datos son confiables ni que tienen la forma esperada. Sin ningún control, un cliente podría enviar campos extra que no deberían modificarse — como el `id` de un libro o campos que no existen en el modelo.

Este tipo de ataque se llama **mass assignment**: el cliente envía un objeto con más propiedades de las esperadas, y el servidor las acepta todas sin filtrar.

Sanitizar significa **extraer explícitamente solo los campos que esperamos** y descartar el resto.

---

### El middleware `sanitizeBookInput`

```typescript
import { NextFunction, Request, Response } from "express"

export const sanitizeBookInput = (req: Request,
 res: Response,
 next: NextFunction) => {
    req.body.sanitizedBookInput = {
        title: req.body.title,
        authors: req.body.authors,
        publisher: req.body.publisher,
        pageCount: req.body.pageCount,
        rating: req.body.rating,
        cover: req.body.cover,
        isAvailable: req.body.isAvailable,
    }

    Object.keys(req.body.sanitizedBookInput).forEach((key) => {
        if (req.body.sanitizedBookInput[key] === undefined) {
            delete req.body.sanitizedBookInput[key]
        }
    })

    next()
}
```

**¿Qué hace exactamente?**

1. **Lista blanca de campos** (*whitelist*): construye un nuevo objeto con solo las propiedades que el modelo `Book` acepta. Cualquier campo extra que el cliente haya enviado (como `id`, `__proto__`, o campos inventados) queda automáticamente excluido porque nunca se copia.

2. **Elimina campos `undefined`**: si el cliente no envió un campo (por ejemplo, en un `PATCH` donde solo manda `rating`), ese campo queda como `undefined` en el nuevo objeto. El `forEach` lo elimina para que `Object.assign` no sobreescriba propiedades existentes con `undefined`.

3. **`next()`**: le indica a Express que este middleware terminó y que puede pasar al siguiente handler en la cadena. Sin esta llamada, el request quedaría colgado y el cliente nunca recibiría respuesta.

---

### Cómo se encadena con los endpoints

En Express, un endpoint puede recibir múltiples funciones en orden:

```typescript
app.post("/api/books", sanitizeBookInput, (req, res) => { ... });
//                     ^^^^^^^^^^^^^^^^^
//                     middleware que corre primero
```

El flujo es:

```
Request POST /api/books
        |
        ↓
sanitizeBookInput (middleware)
  → construye req.body.sanitizedBookInput
  → llama next()
        |
        ↓
Handler principal
  → lee req.body.sanitizedBookInput
  → crea el libro y responde
```

Separar la sanitización en su propio middleware permite **reutilizarla** en `POST`, `PUT` y `PATCH` sin repetir el mismo código en cada handler.
