# Introducción al Código Servidor

## ¿Qué es el Backend?

Una aplicación web moderna se divide en tres capas que trabajan en conjunto:

![3 layers web dev](im%C3%A1genes/3_layers_web_dev.png)

- **Frontend (cliente):** Lo que el usuario ve y con lo que interactúa en el navegador. Está construido con HTML, CSS y JavaScript, y corre en el navegador del usuario.
- **Backend (servidor):** La lógica de negocio de la aplicación. Recibe pedidos del frontend, los procesa, y devuelve una respuesta. El usuario nunca lo ve directamente.
- **Base de datos:** Persiste la información. El backend es el único que se comunica con ella directamente.

La comunicación entre capas sigue este flujo:

```
[Navegador / Frontend]
        |
        |  HTTP Request (ej: "dame todos los libros")
        ↓
[Servidor / Backend]
        |
        |  Consulta SQL / NoSQL
        ↓
[Base de datos]
        |
        ↑  Datos
        |
[Servidor / Backend]
        |
        |  HTTP Response (JSON con los libros)
        ↑
[Navegador / Frontend]
```

El frontend y el backend se comunican mediante **HTTP**: el frontend envía un *request* con un verbo y una ruta, y el backend responde con un *status code* y, generalmente, datos en formato **JSON**.

## Configurando Express

### Inicializar el proyecto

Todo proyecto de Node.js comienza con:

```bash
npm init -y
```


`npm init -y` genera el archivo **`package.json`**, que es el manifiesto del proyecto: describe sus metadatos y lista sus dependencias.

```json
{
  "name": "book-champions-api",
  "version": "1.0.0",
  "description": "",
  "license": "ISC",
  "author": "",
  "type": "module",
  "main": "app.ts",
  "scripts": {
    "dev": "tsc-watch --noClear -p ./tsconfig.json --onSuccess \"node ./dist/app.js\""
  },
  "dependencies": {
    "express": "^5.2.1"
  },
  "devDependencies": {
    "@types/express": "^5.0.6",
    "@types/node": "^25.9.3",
    "tsc-watch": "^7.2.0",
    "typescript": "^6.0.3"
  }
}
```

- **`"type": "module"`** — indica que el proyecto usa ES Modules (`import`/`export`) en lugar de CommonJS (`require`).
- **`dependencies`** — paquetes necesarios en producción (ej: `express`).
- **`devDependencies`** — paquetes solo necesarios durante el desarrollo (ej: TypeScript y sus tipos).

### Instalar dependencias

```bash
npm install express
npm install -D typescript tsc-watch @types/express @types/node
```

Esto descarga los paquetes en la carpeta `node_modules/` y genera el **`package-lock.json`**.

El **`package-lock.json`** congela las versiones exactas de cada paquete instalado (incluyendo dependencias de dependencias). Sirve para garantizar que todos los integrantes del equipo instalen exactamente las mismas versiones al hacer `npm install`.

> **Importante:** `node_modules/` nunca se commitea al repositorio. Se agrega al `.gitignore`. Con el `package.json` y el `package-lock.json` alcanza para que cualquiera pueda recrear el proyecto con `npm install`.

### Scripts

En `package.json` definimos comandos reutilizables bajo `"scripts"`:

```json
"scripts": {
  "dev": "tsc-watch --noClear -p ./tsconfig.json --onSuccess \"node ./dist/app.js\""
}
```

Se ejecutan con `npm run <nombre>`. En este caso: `npm run dev`.

**¿Qué hace este script?**

- **`tsc-watch`** — vigila los archivos `.ts` en busca de cambios y recompila automáticamente cuando detecta uno.
- **`--noClear`** — evita limpiar la consola en cada recompilación.
- **`-p ./tsconfig.json`** — le dice al compilador qué configuración de TypeScript usar.
- **`--onSuccess "node ./dist/app.js"`** — una vez que la compilación termina con éxito, levanta el servidor ejecutando el archivo JavaScript compilado.

El resultado: cada vez que modificamos un `.ts` y guardamos, el servidor se recompila y reinicia automáticamente.

### tsconfig.json

El archivo **`tsconfig.json`** configura el compilador de TypeScript:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "NodeNext",
    "moduleResolution": "NodeNext",
    "rootDir": "./src",
    "outDir": "./dist",
    "sourceMap": true,
    "strict": true,
    "esModuleInterop": true,
    "forceConsistentCasingInFileNames": true,
    "skipLibCheck": true,
    "incremental": true
  },
  "include": ["src/**/*"],
  "exclude": ["node_modules", "dist"]
}
```

| Opción | Qué hace |
|---|---|
| `target` | Define la versión de JavaScript a la que se compila el código TypeScript (`ES2022` es compatible con Node.js moderno). |
| `module` | Especifica el sistema de módulos del código compilado (`NodeNext` usa ES Modules nativos de Node). |
| `moduleResolution` | Define cómo el compilador busca y resuelve los módulos importados. |
| `rootDir` | Carpeta raíz donde viven los archivos fuente `.ts`. |
| `outDir` | Carpeta donde el compilador deposita los archivos `.js` generados. |
| `sourceMap` | Genera archivos `.map` que vinculan el `.js` compilado con el `.ts` original, facilitando el debugging. |
| `strict` | Activa todas las verificaciones de tipo estrictas de TypeScript para detectar errores en tiempo de compilación. |
| `esModuleInterop` | Permite importar módulos CommonJS (como `express`) con la sintaxis de ES Modules (`import express from 'express'`). |
| `forceConsistentCasingInFileNames` | Obliga a que las rutas de importación respeten el mismo uso de mayúsculas/minúsculas que el nombre real del archivo. |
| `skipLibCheck` | Omite la verificación de tipos en los archivos `.d.ts` de `node_modules` para acelerar la compilación. |
| `incremental` | Guarda un caché de la compilación anterior para recompilar solo lo que cambió. |

### Nuestro primer endpoint: `app.ts`

```typescript
import express from "express";

const app = express();

const PORT = 3000;

app.use(express.json());

app.get("/", (req, res) => {
    res.send("You reached the App!");
});

app.listen(PORT, () => {
    console.log(`Server listening in ${PORT}`);
});
```

**¿Qué hace cada parte?**

- **`import express from "express"`** — importa el framework Express.
- **`const app = express()`** — crea la aplicación, el objeto central que recibirá los requests.
- **`app.use(express.json())`** — registra un *middleware*. Un **middleware** es una función que se ejecuta en cada request antes de llegar al endpoint, y puede transformar o inspeccionar el request/response. En este caso, parsea automáticamente el body en formato JSON.
- **`app.get("/", (req, res) => {...})`** — define un endpoint. El callback recibe dos objetos clave:
  - **`req`** (Request): contiene todo sobre el pedido entrante — headers, body, parámetros de ruta, query params.
  - **`res`** (Response): el objeto para construir y enviar la respuesta al cliente.
- **`app.listen(PORT, ...)`** — levanta el servidor y lo pone a escuchar en el puerto 3000.

Al ejecutar `npm run dev`, deberías ver en la terminal:

```
Server listening in 3000
```

Eso confirma que el servidor está corriendo. Podés probarlo abriendo `http://localhost:3000` en el navegador o con una herramienta como [Postman](https://www.postman.com/), [Thunder Client](https://www.thunderclient.com/) o [Bruno](https://www.usebruno.com/).

### ¿Qué es una REST API?

Una **REST API** (Representational State Transfer) es una convención para diseñar endpoints HTTP. En lugar de inventar nombres de acciones (`/getBooks`, `/deleteBook`, `/createBook`), REST aprovecha los **verbos HTTP** y las **rutas** para expresar la intención:

> Un endpoint REST = **verbo HTTP** + **ruta**

Por ejemplo:
- `GET /books` → traer todos los libros
- `POST /books` → crear un nuevo libro
- `GET /books/42` → traer el libro con id 42
- `PUT /books/42` → reemplazar el libro con id 42
- `DELETE /books/42` → eliminar el libro con id 42

La ruta identifica el **recurso** (libro, usuario, etc.) y el verbo indica la **acción** sobre ese recurso.

---

### Verbos HTTP y sus usos

| Verbo | Uso | Tiene body? |
|---|---|---|
| **GET** | Obtener uno o más recursos. No modifica datos. | No |
| **POST** | Crear un nuevo recurso. Los datos van en el body del request. | Sí |
| **PUT** | Reemplazar un recurso completo. Se envía el recurso entero actualizado. | Sí |
| **PATCH** | Modificar parcialmente un recurso. Solo se envían los campos que cambian. | Sí |
| **DELETE** | Eliminar un recurso. | No |

En Express, cada verbo tiene su método correspondiente:

```typescript
app.get("/books", (req, res) => { /* traer libros */ });
app.post("/books", (req, res) => { /* crear libro */ });
app.put("/books/:id", (req, res) => { /* reemplazar libro */ });
app.patch("/books/:id", (req, res) => { /* modificar libro */ });
app.delete("/books/:id", (req, res) => { /* eliminar libro */ });
```

El parámetro `:id` en la ruta es un **parámetro de ruta** (*route param*) que Express captura y pone disponible en `req.params.id`.
