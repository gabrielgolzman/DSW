## Lógica de componentes

Recordemos del apunte 1.0, como es qué React se nos presenta:

>La biblioteca para interfaces de usuario web y nativas

Esto está muy bien pero, ¿No es acaso también posible construir interfaces de usuario con HTML, CSS y JS? Efectivamente, pero lo que nos permiten las librerías como React es construir interfaces de usuario **complejas**, **interactivas** y **reactivas** al usuario, de una manera mucho más simple que sólo con las 3 herramientas nombradas anteriormente.

¿Cuál es el secreto de React para simplificar estos apartados complejos? React utiliza lo denominado **lógica de componentes**. Todas las interfaces de usuario, al final del día, están compuestas por componentes.

**¿Qué son los componentes?** Son bloques de código que funcionan como unidad principal de trabajo en la librería React. Se trabaja con componentes por razones principales:

* **Reusabilidad**: después de crear un componente el mismo puede ser usado en distintas partes del sistema web. Esto nos ayuda a evitar repeticiones.
* **Separación de funcionalidades**: ya que cada componente podrá tener su lógica interna, evitaremos problemas comunes como demasiado código en un solo archivo.

¿Cómo se construyen los componentes? Como toda interfaz de usuario moderna, los componentes están compuestos por HTML5, CSS3 y JS. La diferencia es que React utiliza un acercamiento declarativo en la construcción de estos componentes, donde nosotros definiremos el estado deseado de los elementos en el sitio web, y React se encargará de traducirlos al mismo. Es como si nosotros construyéramos nuestros propios elementos HTML.

## Creando un nuevo proyecto de React
Para la creación de un nuevo proyecto React utilizaremos la herramienta [Vite](https://vite.dev/), con la cuál con una sola línea de comando podremos tener listo un nuevo proyecto, incluyendo las optimizaciones necesarias, la posibilidad incorporada de salida a producción y un servidor de desarrollo local donde podremos ver los cambios que realicemos a nuestro sitio web en tiempo real.

Para poder crear un nuevo proyecto, debemos instalar (o tener instalado) [Node.js](https://nodejs.org/) (versión 20.19 o superior) en la computadora. Esto no es necesario para correr el código (Node.js nos sirve para correr código JavaScript fuera del navegador, pero React funciona en el navegador) sino para poder realizar las instalaciones de los paquetes (como Vite) mediante npm (node package manager).

Vamos a abrir una terminal en la computadora y vamos a dirigirnos a la carpeta donde queremos instalar el proyecto. Cuando estemos allí, correremos el siguiente comando en consola:

```console
npm create vite@latest
```

Nos va pedir el nombre que deseemos para nuestro proyecto.

Luego nos ofrece una serie de opciones: elegiremos **React** como framework y **JavaScript** como variante. El React Compiler no forma parte del scaffolding de Vite y se configura por separado como un plugin (en un futuro apunte, veremos tanto su configuración como el pasaje a TypeScript).

El React Compiler es una herramienta que funciona al momento de *build*, asegurandose la optimización total de nuestra aplicación, evitando así re-renders innecesarios.

Se nos abrirá una página *custom* de bienvenida de React, en caso de que todo haya sido instalado correctamente.

![Página de bienvenida de Vite + React](im%C3%A1genes/Pasted%20image%2020260613175333.png)
*Página de bienvenida de Vite + React tras una instalación exitosa.*

Ahora, vamos a editar el código de esta página para comprobar su funcionamiento. Abrimos la carpeta en VS Code (VSC) (podemos en la misma carpeta abrir una terminal y tipear code . (punto)).

Allí veremos varios archivos:

* ***package.json***: contiene la información de los paquetes instalados (nombre y versionados), junto con las dependencias instaladas.
*  La carpeta **src** (*source*) donde se encuentra el código fuente de toda la página.
* La carpeta **node_modules**, donde están instalados todos los paquetes para que funcione React (y donde se van a instalar las futuras librerías que usemos). El nombre de la misma aparece en gris porque esta carpeta **no debemos** subirla a git.

Realizaremos un poco de limpieza en la carpeta src para comenzar nuestro proyecto:
1. En la carpeta *src*, solo dejaremos los archivos: *App.jsx*, *index.css* y *main.jsx*.
2. Dentro del archivo *App.jsx*, borraremos todo lo que se encuentra dentro del return menos el div principal (borrar su atributo de clase igualmente). Borraremos también ambos *import*.
3. Agregaremos dentro de dicho div un h2 que diga "Books Champion App".

```jsx
const App = () => {
  return (
    <h2>Book Champions App</h2>
  )
}

export default App;
```

**Nota**: usaremos componentes del tipo *arrow function*. Todos los componentes son funciones. A su vez, haremos siempre un sólo *export* por archivo, es decir,  cada componente tendrá su propio archivo.

Para correr el proyecto, debemos utilizar el comando:

```console
npm run dev
```
### Analizando un proyecto de React
Si bien casi siempre hablamos de "este sistema web está escrito en React", "esta aplicación fue construida con React" es importante notar que los componentes **siempre van a estar escrito en JavaScript** (o TypeScript). De React vamos a aprovecharnos de parte de su sintaxis propia y de sus funcionalidades (métodos del ciclo de vida, *hooks*, etc)

El archivo *main.jsx* será el archivo que será servido al servidor y será el que veamos en localhost:5173, sin embargo, el archivo no será servido como está escrito allí, sino que React se encargará de optimizarlo y traducirlo no sólo para que corra en el navegador instalado (como por ejemplo Google Chrome) sino en todo tipo de navegadores junto también a versiones anteriores de lo mismo. 

Buscamos siempre entonces que nuestro sistema web sea **compatible con la mayor cantidad de navegadores y versiones de los mismos**.

Este proceso de traducción y optimización es disparado por el comando *npm run dev*, realizado antes de servir nuestra aplicación en el servidor de desarrollo local. Esto se puede ver en por ejemplo la línea de código:

```jsx
import 'index.css';
```

Ya que es imposible importar un archivo entero de CSS a otro de JS. O por ejemplo en la sintaxis de `<App />`, que tampoco es sintaxis nativa de JavaScript.

Vamos a explicar un poco el siguiente código de *main.jsx*:

```jsx
createRoot(document.getElementById('root')).render(
  <StrictMode>
    <App />
  </StrictMode>,
)
```

Los primeros dos *import* importan al archivo los objetos que representarán a las librerías React y react-dom. Ambas en conjunto forman React, solo que están separadas por cuestiones de funcionalidad. Podemos visualizar que ambas se encuentran listadas en *package.json*.
react-dom va a crear la *root* donde colocará toda la aplicación (en este caso es un elemento del documento *index.html* que tiene la id *root*). Luego en root.render() renderiza la aplicación entera.

Allí, vemos por qué importamos React, en donde dice `StrictMode`, este componente nos permitirá reforzar código limpio y que siga las buenas prácticas de React, dándonos advertencias en consola cuando no logramos hacerlo. Se aplicará sobre todos los componentes que encierre (en este caso, `App`).

Ahora, exploremos el archivo App.jsx:

```jsx
const App = () => {

  return (
  <div>
    <h2>Book Champions App</h2>
  </div>
  )
}

export default App;
```

Vemos que es una función común de JS, llamada App (notemos que no usamos *camelCase* para el nombre de los componentes sino *PascalCase*) que retorna una sintaxis extraña y claramente no válida en JS. Además de eso, vemos que se exporta el componente por *default* (y recordemos de la sección de JS moderno, por eso lo importamos como lo importamos en main.jsx).
Pero, ¿Qué significa esa sintaxis tan extraña y cómo podemos interpretarla?

### JSX
JSX es un acrónimo de JavaScript XML, ya que al fin de cuentas, HTML y XML son descendientes del mismo lenguaje (SGML). Entonces, gracias a React podemos tener código HTML en nuestros archivos JS, ya que esto no funcionará si React no realizará toda la traducción y optimización del código.

**Nota**: Intentaremos llamar JSX siempre que podamos, aunque es común entre los desarrolladores react decirle HTML al JSX retornado por las funciones.

¿Cómo funciona React entonces? El código JSX y la lógica del componente es el estado deseado que queremos ver en el navegador, y React se encarga de presentarlo al servidor.

Si abajo del h2 agregamos la siguiente línea de código:

```jsx
<p>¡Quiero leer libros!</p>
```

Al guardar, chequeamos la página y vemos que automáticamente se agregó el párrafo deseado.
Comparemos lo que nos costaría realizar el mismo proceso en JavaScript normal:

```js
const parag = document.createElement('p'); 
parag.textContext = '¡Quiero leer libros!';
document.getElementById('root').append(parag);
```

Como podemos imaginar, a medida que construyamos sistemas web más complejos, con mayor interacción con el usuario y estados cambiantes (cambio de los estilos, aparición y desaparición de elementos, etc), escribir código de esta manera se volvería tedioso y difícil de llevar seguimiento. 

## Construyendo nuestro primer componente
### Creación
Es práctica común poner los nuevos componentes en su propia carpeta dentro de *src*. Esto se hace para evitar meter todos los componentes al mismo nivel, y de esta forma ser más organizados. El componente `App` si se deja en el lugar que fue creado, ya que es un componente especial al ser el componente raíz, es decir, todos los componentes van a ser hijos de este componente, en forma tipo árbol.

Creamos entonces la carpeta *components*, con la carpeta *bookItem* (cada componente debe encontrarse dentro de su respectiva carpeta en camelCase) y dentro agregaremos un archivo llamado BookItem.jsx. Los componentes, en líneas generales, se suelen llamar con mayúscula inicial en cada palabra (PascalCase).

Dentro de este archivo escribiremos nuestro primer componente, pero, ¿Cómo? Debemos recordar que todos los componentes son funciones (a partir de React 16; la versión actual es React 19) que retornan el jsx que necesitamos. Probemos entonces con:

```jsx
const BookItem = () => {
    return (
        <div>BookItem</div>
    )
}

export default BookItem;
```
Ese componente entonces retorna el texto `BookItem` dentro de un div. Luego, debemos exportar el componente para que lo podamos usar en otros archivos.

```jsx
import BookItem from "./components/bookItem/BookItem"

const App = () => {

  return (
    <div>
      <h2>Book Champions App</h2>
      <p>¡Quiero leer libros!</p>
      <BookItem />
    </div>
  )
}

export default App
```

A `BookItem` lo escribimos simplemente como si fuera una etiqueta cerrada de HTML. Aquí vemos porque son importantes las mayúsculas al momento de nombrar el componente. Sin ellas, React consideraría que es un elemento nativo de HTML (como h2, p, div) y no sabría como interpretarlo, ni hacer la relación con nuestro componente *custom*.

Refrescamos la página y vemos que se ha agregado el componente a la pantalla del usuario.
¡Hemos creado nuestro primer componente! Vamos a agregar más información a `BookItem`.

Al jsx que retorna, agreguemosle un h2 para el título, un h3 para el autor, un div que contenga la puntuación y un elemento p para la cantidad de páginas.

*Error del IDE al intentar retornar múltiples elementos raíz:*

![Error del IDE al retornar múltiples elementos raíz](im%C3%A1genes/Pasted%20image%2020260613183413.png)

Vemos que el IDE nos da error. Esto es debido a una de las reglas más importantes de React: **todo jsx que es retornado debe tener un solo elemento raíz**. En nuestro caso, al tener varios elementos al mismo nivel como raíz, nos da un error. Más adelante, pensaremos una forma más prolija y moderna de solucionar este error, pero por ahora con encerrar todo nuestro código en un div, nos basta.

Podríamos cambiar los contenidos de las etiquetas por algo que refleje un verdadero BookItem, quedándonos así:

```jsx
const BookItem = () => {
    return (
        <div>
            <h2>100 años de soledad</h2>
            <h3>Gabriel García Marquez</h3>
            <div>5 estrellas</div>
            <p>342 páginas</p>
        </div>
    )
}

export default BookItem
```

Vemos que a niveles más adentro del árbol, no hay problemas en poner dos elementos al mismo nivel, es solo la etiqueta raíz la que debe estar sola.
### Mostrando valores dinámicos en pantalla
Como se estará imaginando el estudiante, no sólo tendremos un libro leído en nuestra aplicación y, sobre todo, los datos de ese libro no estarán *hardcoded* en el mismo HTML, sino que sus valores serán dinámicos. Deseamos entonces poseer múltiples componentes con datos que pueden cambiar en el tiempo. Para ello, vamos a crear variables que guardarán los valores actuales de nuestro libro, de la siguiente manera:

```jsx
 const title = "100 años de soledad";
 const author = "Gabriel García Marquez";
 const rating = 5;
 const pageCount = 342;
```

¿Cómo haremos para que React muestre el contenido de nuestras variables en la pantalla? 

Sencillamente, utilizaremos llaves { } , las cuales nos permiten poner código JS que se resuelve en un valor. Por ejemplo, si dentro del h2 colocamos { 1 + 1}, en pantalla nos aparecerá el número 2. Si ponemos { 5>6 ? 'Yes' : 'No' }  esa expresión ternaria se resuelve por No, y es lo que veremos en pantalla. 

Escribiremos el código entonces de la siguiente forma:
```jsx
 <div>
	<h2>{title}</h2>
	<h3>{author}</h3>
	<div>{rating} estrellas</div>
	<p>{pageCount} páginas</p>
 </div>
```

**Nota**: podemos combinar (interpolar) salida común con expresiones JS, como es el caso de la cantidad de páginas.

## *props* en React
### Pasando datos mediante *props*
Ya podemos observar la re-usabilidad de los componentes copiando y pegando `<BookItem />` varias veces, y veremos allí en la pantalla que el componente se multiplica. Si bien esto nos permite utilizar el componente varias veces y en distintas partes, no es realmente "reusable" ya que la *data* que estamos imprimiendo en pantalla es siempre la misma.  

¿Cómo podemos aprovechar al máximo esta funcionalidad entonces? Utilizaremos el concepto similar al de parámetros dentro de las funciones de JavaScript, llamado *props*, (properties) que nos permitirá pasar diferentes valores a los componentes.

Veamos un ejemplo, si tuviéramos el componente App:

```jsx
const App = () => {
 const goalItem = '¡Recibirme!' 
 
 return <CourseGoals/> 
}

```

Y quisiéramos pasárselo al componente CourseGoals:

```jsx
const CourseGoals = () =>
 { 
   return (
   <ul> 	
   	<li>{goalItem}</li>
   </ul>
 ) 
}

```
No podríamos hacerlo directamente, ya que `CourseGoals` no conoce a la variable *goalItem*. Lo que sí podemos hacer es pasarla al componente hijo desde el componente padre, como una propiedad, de la siguiente manera:

```jsx
const App = () => {
 const goalItem = '¡Recibirme!' 
 return  <CourseGoals goal={goalItem}/>
 }

```

Como si fuera un atributo de la etiqueta HTML. Luego, en el componente hijo:

```jsx
const CourseGoals = (props) => {
	return (
		<ul>
			<li>{props.goal}</li>
		</ul>
	)
}

```

**Nota**: el nombre de la *prop* puede ser el mismo que el nombre de la variable, podríamos pasar goalItem={goalItem} y sería válido.

*props* es el objeto que React le pasa a todo componente, conteniendo los atributos que el componente padre le envió. Puede poseer el nombre que queramos pero en general se lo llama props. De esta manera, el componente hijo puede acceder a atributos que posee el componente padre.

Veámoslo en nuestro proyecto:
Vamos a copiar y pegar el componente BookItem siete veces. Luego vamos a crear una constante en "App" llamada `BOOKS` (en mayúsculas, ya que por convención las constantes con datos fijos se escriben en SCREAMING_SNAKE_CASE), que será un arreglo de objetos con siete objetos: nuestro libro que creamos en el componente BookItem y seis más que se nos ocurran. 

Todos los objetos poseerán estos atributos:

* *id*
* *title*
* *author*
* *publisher*
* *rating*
* *pageCount*
* *cover*
* *isAvailable*

Pueden ver los datos de muestra en [MockBooks](MockBooks.md) *(archivo `MockBooks.md` ubicado en la misma carpeta de apuntes)*.

Ahora, pasaremos estos valores a los siete componentes, de esta manera tendremos props dinámicas generadas en JS y no *hardcoded*.

```jsx
  <BookItem
        title={BOOKS[0].title}
        author={BOOKS[0].author}
        publisher={BOOKS[0].publisher}
        rating={BOOKS[0].rating}
        pageCount={BOOKS[0].pageCount}
        cover={BOOKS[0].cover}
        isAvailable={BOOKS[0].isAvailable} />
```
Vemos que no se aplicaron. ¿Por qué es eso? Definitivamente es porque no los declaramos en el componente `BookItem` mediante el objeto *props*. Además de aplicarlo, borraremos la lógica antigua del componente y haremos destructuring sobre props para obtener de manera más limpia las propiedades.

```jsx
const BookItem = ({ title, author, publisher, 
	rating, pageCount, cover, isAvailable }) => {
    return (
        <div>
            <h2>{title}</h2>
            <h3>{author}</h3>
            <div>{rating} estrellas</div>
            <p>{pageCount} páginas</p>
        </div>
    )
}

export default BookItem;
```

### Listas en React
Como podemos imaginar, repetir este proceso manual para cada uno de los siete libros sería tedioso y, en una aplicación real donde la cantidad de datos es desconocida, directamente imposible. Para poder listar componentes en React de manera dinámica, nos valdremos de la función *map*.

Obviamente, no sabemos qué cantidad de libros nos va a devolver el lado servidor para que mostremos en pantalla. Para ello, nos valdremos de la función *map* donde por cada dato, devolveremos un BookItem.
```jsx
 return (
    <div>
      <h2>Book Champions App</h2>
      <p>¡Quiero leer libros!</p>
      {BOOKS.map(book =>
        <BookItem
          key={book.id}
          title={book.title}
          author={book.author}
          publisher={book.publisher}
          rating={book.rating}
          pageCount={book.pageCount}
          cover={book.cover}
          isAvailable={book.isAvailable} />)}
    </div>
  )
```

Observamos allí que agregamos una *prop* llamada *key*. Esta permite identificar unívocamente cada componente de la lista en la UI. Si esta falta, React podría renderizar incorrectamente la lista (confundiendo atributos, por ejemplo). 

**Cada vez que realicemos un map es obligatorio poner la *prop* *key***. Debe ser unívoca.

Como el mapeo es una cantidad de código considerable, lo moveremos a la parte de la lógica del componente.

```jsx
const App = () => {

  const booksMapped = BOOKS.map(book =>
    <BookItem
      key={book.id}
      title={book.title}
      author={book.author}
      publisher={book.publisher}
      rating={book.rating}
      pageCount={book.pageCount}
      cover={book.cover}
      isAvailable={book.isAvailable} />);

  return (
    <div>
      <h2>Book Champions App</h2>
      <p>¡Quiero leer libros!</p>
      {booksMapped}
    </div>
  )
}

export default App
```

Siempre trataremos de mantener lo más limpio posible el JSX retornado por componente, para facilitar su posterior lectura.

## Estilizando en React
### `className` y la *prop* `style`

En React, no podemos usar el atributo `class` de HTML para aplicar clases CSS. Esto es porque `class` es una palabra reservada de JavaScript. En su lugar, React utiliza `className`:

```jsx
// Incorrecto — "class" es palabra reservada de JS
<div class="book-card">...</div>

// Correcto
<div className="book-card">...</div>
```

También existe el *prop* `style`, que acepta un objeto de JS en lugar de una cadena de texto CSS. Las propiedades CSS se escriben en camelCase:

```jsx
<div style={{ backgroundColor: 'red', fontSize: '16px' }}>...</div>
```

Sin embargo, no es una práctica recomendada usar `style` inline para estilos que no cambian dinámicamente. Para estilos estáticos, siempre preferiremos archivos CSS o SCSS externos. `style` sí es la herramienta correcta cuando el valor se calcula en JavaScript y no puede conocerse al momento de escribir el código, por ejemplo:

```jsx
<div style={{ backgroundColor: isAvaliable ? 'green' : 'red' }}>...</div>
```

### SCSS

Para estilizar nuestros componentes de manera más organizada, utilizaremos **SASS** a través de su sintaxis moderna **SCSS**. Para instalarlo en nuestro proyecto:

```console
npm install --save-dev sass
```

Luego, simplemente renombramos nuestros archivos `.css` a `.scss` y actualizamos el import en el componente:

```jsx
import './BookItem.scss'
```

¿Qué ventajas nos da SCSS sobre CSS plano?

* **Anidamiento (*nesting*)**: podemos anidar selectores dentro de otros, lo que hace el código más legible y refleja la estructura del HTML.
* **Variables**: podemos definir valores reutilizables (colores, tamaños, fuentes) con la sintaxis `$variable`:
  ```scss
  $color-gold: #c9a84c;

  .book-card__author {
    color: $color-gold;
  }
  ```
* **El símbolo `&`**: dentro de un selector anidado, `&` representa al selector padre, permitiéndonos construir nombres compuestos sin repetición.

```scss
/* En CSS plano, repetimos el nombre del bloque en cada regla */
.book-card { ... }
.book-card__title { ... }
.book-card__title:hover { ... }

/* Con SCSS, anidamos y usamos & para referenciar al padre */
.book-card {
  ...
  &__title {
    ...
    &:hover { ... }
  }
}
```

### BEM

Para nombrar nuestros selectores CSS/SCSS, utilizaremos la convención **BEM** (*Block, Element, Modifier*):

* **Block** (bloque): el componente en sí. Ejemplo: `book-card`
* **Element** (elemento): una parte del bloque, separada con `__`. Ejemplo: `book-card__title`, `book-card__cover`
* **Modifier** (modificador): una variante del bloque o elemento, separada con `--`. Ejemplo: `book-card__badge--available`, `book-card__star--filled`

```scss
.book-card {               /* Block   */
  &__cover { ... }         /* Element */
  &__badge {               /* Element */
    &--available { ... }   /* Modifier */
    &--reserved  { ... }   /* Modifier */
  }
}
```

¿Por qué usamos BEM? Nos da nombres de clases únicos y descriptivos que evitan colisiones entre componentes. Si tenemos dos componentes distintos, cada uno con su propio bloque BEM, sus estilos nunca se van a pisar entre sí.

**Nota**: el anidamiento de selectores que ofrece SCSS inspiró al estándar CSS, que desde 2024 también lo soporta de forma nativa en todos los navegadores modernos. Sin embargo, SCSS sigue aportando valor adicional a través de variables, mixins y funciones que CSS plano aún no tiene.

## Composición en React
### El patrón de composición

A medida que nuestra aplicación crece, vamos a necesitar componentes de UI genéricos y reutilizables: botones, modales, tarjetas, inputs. Estos componentes no saben de antemano qué contenido van a mostrar — eso lo decide quien los utilice.

Para esto existe el **patrón de composición**: en lugar de crear un `ButtonSave`, un `ButtonCancel` y un `ButtonDelete` por separado, creamos un único componente `Button` flexible que acepta su contenido desde afuera. Para lograrlo, nos apoyaremos en tres herramientas: la prop especial `children`, el operador `...rest`, y los valores por defecto en props.

### *children*

React le pasa automáticamente a todo componente una prop especial llamada `children`, que contiene el JSX colocado **entre las etiquetas de apertura y cierre** del componente:

```jsx
// Quien usa el componente decide el contenido
<Button>Guardar cambios</Button>
<Button>Cancelar</Button>
```

Dentro del componente, simplemente renderizamos `children` donde queramos:

```jsx
const Button = ({ children }) => (
  <button>{children}</button>
)
```

`children` puede ser texto, otro componente, o cualquier JSX válido. Si el componente se usa como etiqueta cerrada (`<Button />`), `children` será `undefined` y simplemente no se renderizará nada en su lugar.

### `...rest` — el operador *spread* en props

Veamos nuestro componente `Button` completo:

```jsx
const Button = ({
  children,
  variant = 'primary',
  size = 'md',
  type = 'button',
  disabled = false,
  onClick,
  ...rest
}) => (
  <button
    className={`btn btn--${variant} btn--${size}`}
    type={type}
    disabled={disabled}
    onClick={onClick}
    {...rest}
  >
    {children}
  </button>
)
```

`...rest` es la sintaxis de *rest parameters* de JavaScript aplicada a la desestructuración. Captura **todas las props que no fueron nombradas explícitamente** y las agrupa en un objeto. Luego, `{...rest}` en el JSX *esparce* ese objeto como atributos del elemento `<button>` nativo.

Esto significa que si quien usa `Button` pasa `aria-label` o `data-testid`, esos atributos llegarán al `<button>` sin que tengamos que declararlos uno por uno:

```jsx
<Button
  variant="secondary"
  aria-label="Guardar cambios"
  data-testid="btn-guardar"
>
  Guardar
</Button>
```

Sin `...rest`, tendríamos que anticipar y declarar explícitamente cada atributo HTML que quisiéramos soportar. Con `...rest`, el componente soporta todos los atributos nativos automáticamente.

### Valores por defecto en props

También podemos observar en `Button` que los parámetros tienen valores por defecto (`variant = 'primary'`, `size = 'md'`, etc.). Esto es simplemente la sintaxis de parámetros con valor por defecto de JavaScript. Si quien usa el componente no pasa esa prop, se usa el valor por defecto:

```jsx
// Usa variant="primary" y size="md" por defecto
<Button>Guardar</Button>

// Sobrescribimos los valores por defecto
<Button variant="danger" size="sm">Eliminar</Button>
```

Combinando `children`, `...rest` y valores por defecto, logramos componentes de UI que son a la vez **flexibles** (soportan muchos casos de uso) y **cómodos de usar** (no requieren configuración mínima para funcionar).
